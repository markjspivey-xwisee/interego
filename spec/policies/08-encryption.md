# Encryption Policy

**Version:** 1.0
**Effective:** 2026-04-25
**Owner:** Operator
**Review cadence:** Annual + after any cryptographic primitive deprecation

## 1. Purpose

To define the cryptographic standards Interego operates under: which algorithms, which key sizes, key lifecycle, and key custody.

## 2. Scope

- All cryptography in `@interego/core` and `@interego/mcp`
- All TLS termination on Azure Container Apps endpoints
- The compliance signing wallet
- Customer-facing identity (DID/SIWE/WebAuthn)
- Pod content encryption (envelope + field-level)
- IPFS content addressing

## 3. Approved Algorithms

| Use | Algorithm | Library |
|---|---|---|
| Symmetric content encryption (envelope payload) | NaCl secretbox (XSalsa20-Poly1305) | tweetnacl |
| Asymmetric envelope key wrap | NaCl box (X25519 + XSalsa20-Poly1305) | tweetnacl |
| Compliance descriptor signature | ECDSA secp256k1 | ethers.js |
| SIWE / wallet signature verification | ECDSA secp256k1 | ethers.js |
| WebAuthn signature verification | ES256 / RS256 / EdDSA | platform |
| Content hashing | SHA-256 | Node crypto |
| IPFS CID hashing | SHA-256 (multihash) | sha2 |
| Merkle tree for ZK aggregates | SHA-256 | implementation |
| TLS | TLS 1.2 minimum, 1.3 preferred; modern cipher suites only | Azure default |

Algorithms not on this list MUST NOT be introduced to the codebase without policy update.

## 4. Policy Statements

### 4.1 Algorithm strength

- Symmetric: 256-bit equivalent strength
- Asymmetric (RSA-equivalent): minimum 2048-bit, recommend 3072-bit; ECC: P-256 / secp256k1 minimum
- Hash: 256-bit minimum

### 4.2 Key generation

- All keys MUST be generated using cryptographically-secure random number generators (Node `crypto.randomBytes` or platform CSPRNG).
- Compliance wallet keys MUST be generated by the operator on a trusted machine (not in cloud-provider browser console).

### 4.3 Key storage

- The active compliance wallet key is stored encrypted at rest in `compliance.wallet.json` using a password-derived key (operator's wallet password).
- The wallet password MUST NOT be checked into source control or stored alongside the wallet file.
- Wallet history file (`compliance.history.json`) holds retired keys for verification of historical descriptors. History MUST be backed up.
- Per-customer envelope keys live in customer pods; operator does not hold them.
- Customer X25519 / SIWE keys are customer-owned; operator never accepts plaintext.

### 4.4 Key rotation

- The compliance wallet SHOULD be rotated annually.
- Immediate rotation MUST occur on suspected key compromise.
- Rotation procedure: `rotateComplianceWallet(walletPath)` — generates new ECDSA key, retires the previous, updates active+history. Old descriptors verify against history; new descriptors signed by the new active key.
- Rotation events MUST publish a compliance descriptor with `dct:conformsTo soc2:CC6.7` containing: rotation timestamp, retired key address, new active key address, reason (scheduled / compromise-response).

### 4.5 Algorithm deprecation

- When NIST or IETF deprecates a cryptographic primitive in use:
  1. Within 30 days, the operator MUST publish a deprecation plan as a compliance descriptor
  2. Within 6 months, replacement primitive SHOULD be implemented
  3. Within 12 months, deprecated primitive MUST be removed (with backward-compatibility shims for verification of historical signatures)

### 4.6 TLS

- All public endpoints MUST use TLS 1.2 minimum, with TLS 1.3 preferred.
- Certificate management is delegated to Azure (managed certs); validity monitored by Azure.
- Certificate expiration MUST page the operator at least 14 days before expiry.

### 4.7 Customer-side cryptography

- The protocol's E2EE design ensures customers control their own envelope keys.
- The operator MUST NOT provide a recovery mechanism for customer-lost keys; this is a structural property (zero-trust storage).
- Customer documentation MUST clearly state: "if you lose your private key, your encrypted content is unrecoverable."

### 4.8 Random number quality

- All RNG usage MUST be CSPRNG-backed.
- Tests MAY use seeded PRNGs for determinism, but production code paths MUST NOT.

## 5. Procedures

### 5.1 Annual wallet rotation

In January each year:

1. Operator runs `rotateComplianceWallet(WALLET_PATH)`
2. Operator confirms history file updated correctly
3. Operator publishes rotation descriptor
4. Operator validates a new compliance descriptor signs + verifies under new active key
5. Operator validates an old descriptor still verifies via history walk

### 5.2 Wallet backup

Daily automated backup of `compliance.wallet.json` and `compliance.history.json` to encrypted off-site storage. Backup location documented separately (operator's secure notes).

### 5.3 Compromise response

If wallet compromise is suspected:

1. Within 1 hour: rotate wallet
2. Within 4 hours: publish compromise notification descriptor
3. Within 24 hours: investigate scope (which descriptors during which window are now untrusted)
4. Within 7 days: post-mortem published

## 6. Exceptions

Cryptographic exceptions are categorically prohibited. Any deviation from approved algorithms requires policy update before implementation.

## 7. Mapping

- SOC 2: CC6.7 (transmission and storage encryption)
- ISO 27001 Annex A: A.10.1
