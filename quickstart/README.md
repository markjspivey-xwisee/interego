# Interego quickstart — substrate-up in under 10 minutes

This directory bootstraps a working Interego deployment with two
ready-to-use surfaces:

- **personal-bridge** at `http://localhost:5050` — your local-first
  Tier-5 substrate (embedded relay, MCP over HTTP, REST + admin UI,
  identity rooted in a wallet you control).
- **organizational-working-memory bridge** at `http://localhost:6060`
  — a reference vertical that demonstrates how a "company memory"
  product is recoverable from Interego's primitives alone (typed
  descriptors, content-addressed atoms, modal status, supersedes
  chains, per-source navigation).

Both surfaces speak Streamable-HTTP MCP, so any MCP-aware agent
(Claude Code, Claude Desktop, Cursor, Cline, custom) drops in by
config alone.

## 1. Pick a pod

Two equally valid choices:

**A. Hosted reference pod** — fastest evaluation, zero infrastructure.
   The maintainer runs an Azure CSS instance for evaluation. You'll
   write to a sub-container under your assigned user prefix:
   `https://interego-css.livelysky-8b81abb0.eastus.azurecontainerapps.io/u-pk-<your-id>/`

**B. Your own Solid pod** — production path. Any CSS-compatible pod
   works. The bridge writes to `<pod>/context-graphs/`.

## 2. Configure

```bash
cp .env.example .env
$EDITOR .env       # set OWM_POD_URL at minimum
```

The only required value is `OWM_POD_URL`. Everything else is optional
or autogenerated.

## 3. Boot

From this directory:

```bash
docker compose up --build
```

You should see, within ~30s:

```
interego-personal-bridge  | personal-bridge ready on :5050
interego-owm-bridge       | organizational-working-memory bridge on http://localhost:6060
interego-owm-bridge       |   Source adapters:     web
interego-owm-bridge       |   10 affordances available
```

## 4. Wire your MCP client

Copy [`mcp-config.example.json`](mcp-config.example.json) into your
client's MCP configuration (or merge the `mcpServers` entries). For
Claude Code:

```bash
claude mcp add-json --scope user personal-bridge '{"type":"http","url":"http://localhost:5050/mcp"}'
claude mcp add-json --scope user owm-bridge      '{"type":"http","url":"http://localhost:6060/mcp"}'
```

Then start Claude Code anywhere; both bridges are reachable.

## 5. First useful interaction (~2 minutes)

In a fresh Claude Code session, ask:

> *"Using the owm-bridge tools, fetch https://www.rfc-editor.org/rfc/rfc7232.html via owm.navigate_source, distill two contributors as owm.upsert_person, create an owm.upsert_project for HTTP conditional-request conformance, record a Hypothetical decision about adopting If-Match for our publish() manifest CAS, and queue a follow-up due tomorrow."*

When the agent reports back, you'll have:

- 2 typed `owm:Person` descriptors on the pod (signed by the org DID).
- 1 `owm:Project` with an OLKE knowledge stage.
- 1 `owm:Decision` with `cg:modalStatus = Hypothetical`.
- 1 `owm:FollowUp` with a future `owm:dueAt`.
- A `cg:supersedes` chain ready to receive future revisions.

In a SECOND fresh session — no shared state with the first — ask:

> *"Use owm-bridge to list any overdue follow-ups and walk the project's subgraph."*

The Surfacer recovers Phase 1's state from the pod alone. That's
the proof: the substrate is the audit trail.

## 6. What's next

- **Demos.** [`../demos/`](../demos/) ships fifteen end-to-end
  scenarios driving real claude processes against real bridges.
  `./demos/run-all.sh 15` runs the OWM scenario; the others
  exercise modal-status flips, PGSL pullbacks, parallel writers,
  cryptographic citation chains, multi-agent teaching transfer,
  constitutional democracy, and ZK selective disclosure.
- **More verticals.** Each one under [`../applications/`](../applications/)
  is a separate package on its own port. They are intentionally
  independent — verticals do not bundle into the generic personal-
  bridge or the hosted relay.
- **More source adapters.** OWM ships with a `web` adapter; the
  framework at [`../applications/organizational-working-memory/source-adapters/`](../applications/organizational-working-memory/source-adapters/)
  is ~50 lines to add a new one (drive, slack, github, gmail, ...).
  Each adapter runs as an isolated sub-handler so the main agent's
  context never sees the per-source surface.

## Troubleshooting

- **Port already in use** — kill leftover bridges from previous runs.
  On Windows: `netstat -ano | findstr ":5050 :6060"` then
  `taskkill /F /PID <pid>`. On macOS / Linux: `lsof -i :5050 -i :6060`
  then `kill <pid>` or `pkill -f "tsx server.ts"`.
- **OWM bridge exits with "OWM_POD_URL must be set"** — fill in
  `quickstart/.env`. Pod selection is intentionally explicit so
  the bridge doesn't silently write to a default it shouldn't.
- **MCP client doesn't see the tools** — confirm both bridges respond
  to `curl http://localhost:5050/health` and `curl http://localhost:6060/`.
  The OWM bridge advertises its tool list at `POST /mcp`
  (`tools/list` JSON-RPC method).
